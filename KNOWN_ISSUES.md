# TTS 语音助手 - 已知问题与限制

> **适用版本**: MVP v0.9.0  
> **本文档性质**: 诚实记录当前版本的已知限制和待改进项，帮助用户建立合理预期。

---

## 一、功能限制

### 1. 全局热键已集成（Day 10 更新）

- **现状**: `src/core/hotkey_manager.py` 中的全局热键底层实现已改为 **Windows 原生 `RegisterHotKey`**（通过 `ctypes` + `QAbstractNativeEventFilter`），与 PyQt6 的 `QSystemTrayIcon` 100% 兼容，彻底解决了早期 `pynput` 方案导致的托盘点击失效问题。默认热键为 `Ctrl+Shift+T`。
- **注意**: 热键消息直接在 Qt 主线程事件循环中拦截，无需额外监听线程，也不会与托盘消息泵冲突。
- **自定义**: 可在 `config.json` 中修改 `"hotkey"` 配置项（如 `"<alt>+r"`、`"<ctrl>+<alt>+t"`）。支持 `ctrl`、`alt`、`shift`、`cmd`（Win 键）与字母/数字/F1-F12 的组合。
- **限制**: 若与其他软件的全局热键冲突，Windows 会返回注册失败，此时仍可通过托盘唤出窗口。

### 2. 游戏全屏覆盖可能失败

- **现状**: 悬浮窗使用 `WindowStaysOnTopHint` 实现全局置顶。
- **限制**: 部分全屏游戏（尤其是使用独占全屏模式的 3A 大作）会接管 Windows 的顶层窗口管理权，导致第三方悬浮窗无法显示在游戏画面上。
- **workaround**: 在游戏中将显示模式切换为**无边框窗口（Borderless Windowed）**或**窗口化全屏**，悬浮窗即可正常覆盖。
- **计划**: 此限制属于 Windows 安全策略范畴，暂无完美解决方案，后续会尝试使用更底层的注入或 overlay 技术（风险较高，优先级低）。

### 3. 单次文本长度限制 500 字

- **现状**: 程序会自动截断超过 500 个字符的输入文本。
- **原因**: edge-tts 对超长文本的生成稳定性较差，且游戏场景下的 TTS 以短句为主。
- **workaround**: 如需朗读长文，请手动分段输入。
- **计划**: v0.2 支持自动长文本分段与合并播放。

### 4. 不支持真实麦克风与 TTS 混音

- **现状**: TTS 语音直接输出到 VB-CABLE 虚拟声卡。
- **限制**: 当语音软件使用 CABLE 作为麦克风时，你自己的真实麦克风将无法同时被使用。
- **workaround**: 如需"自己说话 + TTS"同时被队友听到，需要使用 VoiceMeeter Banana 等第三方虚拟混音台软件，将真实麦克风和 VB-CABLE 混音后再输出。
- **计划**: 暂不考虑在工具内部实现混音功能（会显著增加复杂度和资源占用）。

## 二、兼容性问题

### 1. 杀毒软件误报

- **现状**: 默认打包为 `--onefile` 单文件 exe（约 60MB）。
- **限制**: 360、火绒、Windows Defender 等安全软件可能将 PyInstaller 打包的单文件 exe 识别为可疑程序（ heuristic detection ）。
- **workaround**:
  1. 在杀毒软件中将程序添加为信任/白名单
  2. 使用 `build.py` 的 `--onedir` 模式重新打包（目录版误报率显著降低）
  3. 直接从源码运行，彻底避免误报
- **计划**: 后续考虑代码签名证书（需成本）进一步降低误报率。

### 2. VB-CABLE 为必需依赖

- **现状**: 音频转发完全依赖 VB-CABLE 虚拟声卡驱动。
- **限制**: 未安装该驱动的用户无法将 TTS 语音输出到语音软件。
- **workaround**: 按照 [INSTALL.md](./INSTALL.md) 的说明，从官网下载并安装 VB-CABLE 驱动。
- **计划**: 评估集成其他虚拟音频方案（如 Virtual Audio Cable 付费版）的可能性，但 VB-CABLE 仍将是主推方案。

### 3. 仅支持 Windows 10/11

- **现状**: 使用了 Windows 原生 `RegisterHotKey`（全局热键）、pyttsx3（依赖 Windows SAPI5）和 VB-CABLE（Windows 驱动）。
- **限制**: 不支持 macOS 和 Linux。
- **计划**: MVP 阶段聚焦 Windows 游戏场景，跨平台支持不在短期计划中。

## 三、网络与音质

### 1. edge-tts 受网络环境制约

- **现状**: 主引擎 edge-tts 需要访问微软的在线语音合成服务。
- **限制**: 在网络不稳定、DNS 污染或微软服务被墙的区域，edge-tts 可能连接失败或响应缓慢（超时阈值设为 5 秒）。
- **workaround**: 程序会自动降级到系统 TTS（pyttsx3/SAPI5），保证离线可用；常用语在缓存命中后也可完全离线播放。
- **计划**: v0.3 计划引入 Piper 本地 TTS 模型，实现完全离线的高质量语音合成，彻底摆脱网络依赖。

### 2. 系统 TTS 降级音质为"机械音"

- **现状**: 降级引擎使用 Windows 内置的 SAPI5 语音。
- **限制**: 中文语音包（如 Microsoft HuiHui）音质明显不如 edge-tts 自然，听起来像"机器人"。
- **workaround**: 保持网络通畅，让程序优先使用 edge-tts；多使用常用语以命中本地缓存。
- **计划**: 同上，引入 Piper 本地模型后可解决。

### 3. 首次播放新文本有 0.5~2 秒延迟

- **现状**: edge-tts 需要通过网络请求生成 MP3 音频文件。
- **限制**: 首次生成新文本时，延迟主要取决于你的网络速度（通常 300ms ~ 2000ms）。
- **workaround**: 缓存命中后延迟降至 < 200ms；程序启动时会后台预加载 7 条常用语。
- **计划**: v0.2 引入模糊匹配缓存，提升近似文本的缓存复用率。

## 四、性能与稳定性

### 1. 内存占用估算

- **现状**: 实测内存占用约 50~80MB。
- **限制**: 长时间运行后，由于 Python 内存管理机制，内存占用可能会缓慢上升至 100MB 左右，但通常不会持续增长。
- **workaround**: 如感觉系统变卡，可右键托盘图标选择"退出"后重新启动。
- **计划**: 持续优化音频句柄和 pygame mixer 的释放逻辑。

### 2. 配置文件为明文 JSON

- **现状**: 配置存储在 `config.json` 中，为纯文本格式。
- **限制**: 不含敏感信息，但用户可直接编辑，若格式写错可能导致程序启动异常。
- **workaround**: 如程序启动异常，删除 `config.json` 即可恢复默认配置。
- **计划**: 后续增加 JSON 格式校验和自动恢复机制。
